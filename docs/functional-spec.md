# ASUTPORT — сводная функциональная спецификация

Единый портал технической поддержки для рынка АСУ ТП РФ — для всего спектра оборудования и ПО: ПЛК (железные и виртуальные), SCADA, панели оператора, частотные приводы, датчики и КИП, промышленное сетевое оборудование, MES-прослойки. Виртуальный ПЛК в примерах ниже — первый пилотный кейс (на софтовых продуктах механика платформы демонстрируется в полную силу дешевле всего), а не специализация платформы. Пять ролей: клиент (эксплуатация), производитель, поставщик/дилер, интегратор, суперадмин (платформа). Документ сведён по итогам проработки модели: юнит-экономика v2 (маркетплейс + ИИ-линия, без собственного штата поддержки), трёхролевая маршрутизация по entitlement, фолбэки и монетизация прикладной зоны.

---

## 0. Сквозные механизмы

Это не функции отдельной роли, а ядро, на которое опираются все кабинеты. Вынесены отдельно, чтобы не повторяться, — в разделах ролей на них есть ссылки.

### 0.1 ИИ-агент (L0/L1)

Первая линия всей платформы. Обучен на документации всех подключённых производителей (vision-пайплайн: страницы парсятся как изображения через VLM, таблицы сохраняются как HTML, формулы как LaTeX, схемы получают текстовые описания; картинка страницы хранится рядом с каждым чанком и подаётся мультимодальной модели при ответе — страховка от ошибок парсинга). Ключевые свойства:

- **Отвечает со ссылкой на конкретную страницу мануала.** Проверяемость — механизм доверия инженерной аудитории; ответ без источника не принимается.
- **Версионность.** Каждый фрагмент базы знаний несёт метку версии продукта; ретривал фильтруется по версии из профиля установки. Ответ по чужой версии хуже отсутствия ответа.
- **Адаптивный опрос вместо анкеты.** Агент спрашивает только то, что релевантно симптому, и каждым вопросом разводит конкурирующие гипотезы (пример: «падает ли на пустом проекте?» разводит дефект и прикладную ошибку).
- **Честная неопределённость.** При уверенности ниже порога — «в документации этого нет» и предложение эскалации, а не догадка. Пороги асимметричны по цене ошибки: лишняя эскалация дёшева, ложное «это ошибка вашего кода» — дорого.
- **Классификация уточняющаяся, не одноразовая.** Тип тикета фиксируется к моменту эскалации; переквалификация инженером записывается в датасет для измерения и улучшения точности.
- **Ограничения по политике.** Агент никогда сам: не выносит вердикт «это дефект» (только производитель), не принимает гарантийно-денежных решений (только поставщик), не даёт рискованных инструкций на аварийном приоритете непрерывного производства без параллельного предложения эскалации.

Категории, которые агент закрывает сам (целевые 50–70% потока): «как настроить/как сделать»; известные проблемы (матчинг с решёнными тикетами и статьями); диагностика до вердикта (сбор логов, слепок, воспроизведение — даже если тикет потом уйдёт наружу, 80% работы сделано); справки по entitlement (гарантия, тариф, лицензии — чтение записи без рассуждений); совместимость и подбор (включая карты замены ушедших западных вендоров — лидоген для производителей); прикладной код в режиме best-effort.

### 0.2 Профиль установки и entitlement

Профиль установки — описание «зоопарка» клиента: продукты, версии, среда (ОС, виртуализация, протоколы), тип производства (критичность), разрешение на диагностическую утилиту, аварийный контакт. Заполняется один раз визардом при онбординге; перед каждым обращением агент лишь подтверждает дельту одним вопросом («что-то меняли?»), и сама дельта — диагностический сигнал (половина инцидентов начинается «после обновления»).

Entitlement — записи уровня **экземпляра** (серийник / лицензионный ключ): производитель, поставщик, дата покупки, срок гарантии, номер договора, интегратор проекта, тариф поддержки. Источники по убыванию надёжности: реестр производителя (верифицированная запись, заодно фильтрация серого рынка), загрузка отгрузок поставщиком, слова клиента при онбординге (с пометкой). Правило маршрутизации: тип вопроса — вывод агента, entitlement — справочник, превращающий вывод в конкретного адресата. LLM рассуждает, данные маршрутизируют; местами не меняются.

Накопленный побочный актив: карта «где какое оборудование стоит и кто за него отвечает» — на рынке РФ её нет ни у кого.

### 0.3 Маршрутизация тикетов и SLA-трекинг

Клиент никогда не выбирает адресата — единое окно. Маршруты по типу: типовой вопрос → агент; дефект продукта → производитель (всегда: только у него код, знание, дедупликация и стимул чинить); гарантия/лицензии/замена/докупка → поставщик конкретного экземпляра по entitlement; прикладная логика → интегратор проекта. Стыковой случай (симптом на границе двух продуктов) → связанный тикет обоим производителям.

Стыковые тикеты держатся на четырёх опорах: контрактное обязательство каждого производителя отвечать за свою часть диагностики (узкое: не чинить чужое, а дать заключение по своему); платформа приходит с уликами со стенда, а не с «разберитесь сами»; общий коммерческий интерес в работающей связке (бейдж «проверенная связка» — аргумент обоих в тендерах); прецедент TSANet как доказательство работоспособности модели. Фолбэк-режим для несогласных: асимметричный (клиент видит один тред, под капотом два тикета, супервайзер синхронизирует вручную).

SLA-трекинг: приоритеты (авария / деградация / вопрос), живые таймеры, статус «на чьей стороне мяч», автоэскалация по цепочке (инженер → руководитель поддержки вендора → супервайзер платформы). SLA клиенту формулируется честно: «время реакции платформы + время решения для подключённых сторон»; на внешнюю неподключённую часть гарантия не даётся и это видно в кабинете (покрытие продуктов: полное/частичное).

Фолбэк «нужной стороны нет на платформе»: пустая запись → агент сначала спрашивает; запись есть, компании нет → по типу тикета: гарантийка — контакты поставщика + готовое обращение со всей диагностикой; прикладное — ступени услуг платформы (см. 0.5); дефект без производителя (ушедшие западные вендоры) — платформа как «производитель поддержки» для легаси: база знаний, комьюнити, биржа. Каждый фолбэк логируется как growth-сигнал (см. роль суперадмина). Метрика здоровья: доля фолбэков (в первый год 40–50% — нормально, но фолбэк обязан быть самообслуживаемым, иначе экономика ползёт к модели v1 с штатом).

### 0.4 База знаний

Публичная (SEO-воронка: инженер гуглит ошибку ночью → статья → виджет «спроси агента» → регистрация). Состав: официальная документация производителей, статьи из решённых тикетов (анонимизированные; категория «как настроить» обезличивается тривиально), стыковые материалы, которых нет ни у одного вендора по отдельности («ПЛК X + SCADA Y на Astra»), карты миграции с ушедших вендоров. Контур самоусиления: каждый решённый тикет → статья → агент дообучен → следующее такое обращение закрывается без людей. Это же главный защитный актив против копирования.

### 0.5 Стенд и диагностика

Диагностическая утилита у клиента: запускается вручную, по запросу, ничего не передаёт без действия пользователя (промышленная паранойя учтена в UX и в тексте согласия). Глубина слепка зависит от класса оборудования: для софтовых продуктов (виртуальный ПЛК, SCADA) — полный снимок: версии, параметры каналов, логи за час; для железных ПЛК — выгрузка проекта + системный журнал; для периферии (приводы, датчики), чья конфигурация живёт в инструменте наладки или проекте контроллера, — выгрузка проекта + опрос устройств по шине. Обещания клиенту калиброваны по классу — автоматизация разной глубины, честно заявленной.

Воспроизведение («машина по производству улик») — лестница методов от дешёвого к дорогому: (1) софтовые продукты — авторазвёртывание копии связки клиента в ВМ по слепку, почти бесплатно (поэтому пилот с вендором виртуального контроллера показывает механику платформы в полную силу); (2) железо — эмуляция протокольного поведения (Modbus/OPC UA-слейв, воспроизводящий устройство по логам клиента — дёшево, покрывает большинство кейсов уровня обмена); (3) физический стенд с ходовым железом — не своё, а партнёрство с дистрибьюторами, у которых демо-лаборатории уже есть; (4) воспроизведение на стороне производителя как финальная ступень. Принцип неизменен для всех классов: эскалация уходит с уликами, а не с «у клиента что-то не работает».

Прикладная зона (осиротевшие проекты) — трёхступенчатая: (1) агент читает прикладной код (ST/LD), типовые болячки — best-effort без SLA с дисклеймером про процедуру изменений на работающем производстве; (2) «паспорт осиротевшего проекта» — платная реконструкция документации по выгрузке (карта задач, теги, логика блоков словами), чек 50–300 тыс; (3) биржа специалистов — матчинг по стеку, эскроу, репутация, комиссия 10–15%; специалист входит в проект с готовым паспортом.

## 1. Клиент (эксплуатация / интегратор-в-роли-пользователя)

Персона: инженер АСУ ТП на объекте, служба КИПиА, инженер эксплуатирующего интегратора. Путь: гуглит ошибку → статья базы знаний → агент → бесплатная регистрация → конверсия в платный тариф в момент боли (висящий тикет + кнопка «с Priority реакция 2 часа»); счёт на юрлицо в два клика.

**Функционал кабинета:**

1. **Профиль установки** (см. 0.2) — визард при онбординге, редактирование, несколько партий/поставщиков на один продукт, привязка серийников. Подтверждение дельты перед обращением.
2. **Чат с агентом** — полный доступ по подписке, ограниченный без регистрации. Всё из 0.1: адаптивный опрос, ссылки на страницы, персонализация под профиль («в вашей конфигурации с кластером учтите…»), честные отказы.
3. **Единое окно тикетов** — создание без выбора адресата; список и деталка с лентой событий, «мяч на стороне», живой SLA-таймер, автоэскалация; общий тред стыкового тикета с видимой диагностикой обеих сторон.
4. **История инцидентов установки** — накопленная память: «похожая проблема была в марте, тогда помогло…» (слой памяти по типу Mnemoniqa). Решённые тикеты со ссылками на порождённые статьи.
5. **Диагностическая утилита** — слепок по запросу (глубина по классу оборудования, см. 0.5), приложение к тикету одним действием; согласие на воспроизведение на стенде платформы, где применимо.
6. **База знаний** — поиск по всем вендорам установки, стыковые статьи, фильтр по своим версиям.
7. **Покрытие и тарифы** — видимость, какие продукты установки покрыты полностью (все стороны на платформе), какие частично (фолбэк); тарифная лестница: бесплатный (агент + база знаний, тикеты без гарантий) → входной ~25 тыс/мес (приоритет в очереди) → Priority ~60 тыс/мес (SLA реакции, 8/5) → 24/7 (вводится только при 5+ клиентах по 180+ тыс — иначе дежурные смены съедают экономику). Лимит тикетов в тарифе с доплатой за превышение — зашит в оферту с первого дня (защита экспертов вендоров и экономики).
8. **Заказ услуг прикладной зоны** — анализ кода, паспорт проекта, подбор специалиста с биржи (см. 0.5). Поддержка легаси ушедших вендоров — как отдельное предложение с отдельным питчем (боль «Siemens не отвечает» острее любой боли по российским продуктам).

## 2. Производитель

Персона внутри: инженер поддержки/разработки (живёт в очереди эскалаций), продакт-менеджер (живёт в аналитике), сейлз (использует портал в тендерах), руководитель поддержки (получатель автоэскалаций). Путь: пилот 3 мес бесплатно (документация + архив обращений → агент → «экзамен» инженером → приглашение пользователей) → дашборд с цифрами → конверсия по заранее зафиксированной цене при закрытии агентом >40–50%. Тарифы: базовый ~100–120 тыс/мес, расширенный ~300 тыс (аналитика, брендированный раздел, SLA-эскалации). Для молодого вендора без поддержки — замена найма; для гранда — витрина + разгрузка + стыковые.

**Функционал кабинета:**

1. **Очередь эскалаций** — только то, что агент не закрыл; каждая приходит квалифицированной: сводка агента, логи, слепок, шаги воспроизведения, воспроизведение на стенде, история похожих обращений. Дефекты приходят дедуплицированными («7 воспроизведений на v2.3.x») и напрямую, минуя испорченный телефон дилеров.
2. **Работа с тикетом** — ответ клиенту (переводит мяч, закрывает SLA реакции), передача коллеге с сохранением таймера, переквалификация типа (пишется в датасет точности), участие в стыковых тредах в рамках контрактного обязательства «заключение по своей части».
3. **Зона поддержки (политика)** — декларация границ ответственности, а не право отказываться от тикетов: матрица версий (полная поддержка / только критичные / EOL best-effort), граница «продукт vs прикладной код», часы покрытия и приоритеты (read-only зеркало контракта с кнопкой «изменить тариф»). Машиночитаема — агент применяет её до создания тикета (ожидание клиенту выставляется ещё в чате: «версия EOL, вот миграция + best-effort»). Публична на странице вендора — рынок сам наказывает слишком узкую зону. Чего в настройках нет: выхода из стыковых, паузы SLA, скрытых отказов. На MVP — YAML при онбординге, UI позже.
4. **Маршрутизация внутрь команды** — темы → конкретные инженеры, календарь дежурств, обязательный резервный контакт.
5. **Редактор базы знаний** — правки дообучают агента; черновики статей из решённых тикетов на утверждение («ответили один раз — агент отвечает всегда»); советы агента («31 обращение про Modbus на версиях до 2.3 — чек-лист снизит поток на четверть»).
6. **Аналитика** — топ тем, распределение по версиям, динамика доли закрытого агентом, сэкономленные часы, медиана реакции; продуктовые сигналы для роадмапа. Ежемесячный отчёт (на пилоте — главный инструмент конверсии).
7. **Реестр серийников/лицензий** — сверка entitlement: верификация официальных поставок, фильтрация серого рынка, впервые — видимость собственной инсталлированной базы (где реально стоят экземпляры).
8. **Витрина и лиды** — публичная страница с зоной поддержки и временем реакции, брендированный раздел (расширенный тариф), лиды («смотрели документацию, спрашивали про замену CODESYS»), бейдж «проверенная связка» по работающим стыковым парам, строка для тендерной заявки «поддержка через отраслевой портал с SLA» — закрывает главное возражение против молодого вендора.
9. **Публичный рейтинг времени ответа** — агрегированное время реакции по вендорам видно всем: мягкое давление, делающее SLA выполнимым без штата платформы.

## 3. Поставщик / дилер (роль «Партнёр»)

Персона: менеджер дилера, сервис-инженер дистрибьютора. Их у одного продукта десятки — неоднозначности нет, потому что гарантийный тикет маршрутизируется по экземпляру (серийник → партия → конкретный поставщик). Для железного оборудования доля гарантийно-коммерческих тикетов (замена, брак экземпляра, поставка) существенно выше, чем у софта, — поэтому по мере расширения платформы с софтовых продуктов на весь спектр АСУ ТП партнёрская роль и entitlement по серийникам из «третьей ноги на потом» становятся ядром механики. Подписка дешевле вендорской: ~20–40 тыс/мес. Канал продаж роли — отчёты о фолбэках: «за квартал 11 обращений ваших клиентов прошли мимо вас» (продажа случившегося факта, не абстракции). Воронка ролей шире вендорской: производителей ~20, дилеров и интеграторов — сотни.

**Функционал кабинета:**

1. **Тикеты своих клиентов** — только по своим записям entitlement: гарантия, замена, брак экземпляра (агент различает «ошибка продукта» и «ошибка экземпляра» — второе зона дилера), лицензии, продления, апгрейды, докупка. Структурированно, с уже собранными агентом серийниками и диагностикой.
2. **Загрузка отгрузок** — «этому клиенту такие серийники»: наполняет entitlement; прямой стимул — иначе гарантийные тикеты клиентов уходят в фолбэк мимо дилера, и он теряет и время реакции, и видимость.
3. **Видимость проблем своей клиентской базы** — все тикеты своих клиентов (в рамках приватности), здоровье установок.
4. **Сигналы апсейла** — «у клиента три тикета про нехватку производительности — время предложить апгрейд»; переток решённых техпроблем в коммерческие возможности.
5. **Позиционирование относительно первой линии** — платформа не отбирает у дилера первую линию (частое договорное обязательство дилера перед производителем), а выполняет её лучше: типовое закрывает агент, дилеру остаётся законная гарантийно-коммерческая зона плюс видимость, которой у него не было.

## 4. Интегратор

Персона: инженер/руководитель проектов интегратора. Может совмещать роли: интегратор часто и поставщик (продал оборудование в составе проекта — тогда в entitlement он стоит в двух полях, маршрутизация не меняется), и клиент (эксплуатирует сданные объекты). Стратегическая рамка: сироты — зона платформы, живые проекты — зона интеграторов; платформа — источник заказов, не конкурент.

**Функционал кабинета:**

1. **Тикеты прикладной логики своих проектов** — маршрутизируются по полю integrator в entitlement; приходят с диагностикой агента и паспортом проекта, если он есть.
2. **Реестр своих проектов** — привязка к установкам клиентов, актуальность контактов (защита от превращения в «исчезнувшего интегратора» в глазах платформы).
3. **Лиды на модернизацию** — платформа передаёт живым интеграторам всё, что крупнее диагностики на осиротевших и чужих проектах их профиля («проект вашего стека, нужна модернизация — берёте?»).
4. **Биржа — сторона исполнителя** — инженеры интегратора как специалисты: профили по стеку, репутация по выполненным заказам, эскроу; вход в осиротевший проект с готовым паспортом (снижает худший риск заказа «чужой код без документации»).

## 5. Суперадмин (платформа)

Персона: ты + супервайзер (0,5–1 FTE «играющий тренер»). Принцип экономики v2: платформа не нанимает людей для ответов — люди-эксперты только у вендоров; в штате арбитраж, контроль качества и рост. Всё, что ниже, — инструменты, которые держат эту модель.

**Функционал:**

1. **Онбординг производителя** — приём документации и архива обращений, прогон vision-пайплайна, сборка агента по продукту, «экзамен» инженером вендора, золотой набор из 30–50 злых вопросов по таблицам/схемам/формулам с эталонами — прогоняется после каждого изменения пайплайна (и служит артефактом пилота: «92% по вашей карте регистров, вот прогон»). Настройка зоны поддержки (YAML) и маршрутизации команды.
2. **Арбитраж стыковых тикетов** — роль арбитра в общих тредах; ручная синхронизация асимметричного режима для вендоров, не готовых к общим тредам; фиксация «на чьей стороне мяч».
3. **Управление SLA-политиками** — приоритеты, нормативы по тарифам, цепочки автоэскалаций, неустойки/обязательства в договорах с вендорами; контроль честности формулы «реакция платформы + решение подключённых сторон».
4. **Метрики здоровья платформы** — панель: доля закрытого агентом (главный рычаг маржи и оффера: сдвиг 50→70% почти удваивает ёмкость), доля фолбэков (следить за самообслуживаемостью), точность квалификации (по датасету переквалификаций), время реакции вендоров (питает публичный рейтинг), нагрузка на клиента (лимиты тарифов), unit-метрики из калькулятора (вендоры/клиенты/чеки/точка безубыточности).
5. **Качество агента** — пороги уверенности (асимметричные по цене ошибки), правила-ограничители (вердикты о дефектах, деньги, аварийные приоритеты), контроль дообучения на новых статьях и тикетах, регрессионные прогоны золотых наборов всех вендоров.
6. **Модерация базы знаний** — анонимизация тикетов в статьи, утверждение черновиков, стыковые статьи (пишутся платформой — это её уникальный контент), карты миграции с ушедших вендоров.
7. **Entitlement-администрирование** — разрешение грязных случаев (серые поставки: честный ответ клиенту + защита официального канала производителей; исчезнувшие поставщики: перенос гарантии; «клиент не помнит»: техчасть сразу, запись потом), сверки с реестрами, качество карты установок как актива.
8. **Стенд** — управление лестницей воспроизведения (см. 0.5): парк ВМ с авторазвёртыванием для софтовых продуктов, библиотека протокольных эмуляторов для железа, партнёрства с демо-лабораториями дистрибьюторов, библиотека воспроизведений, приоритизация стендового времени под аварийные тикеты.
9. **Биллинг и тарифы всех ролей** — клиентские планы с лимитами, вендорские подписки, партнёрские, комиссия биржи, разовые услуги прикладной зоны (паспорт проекта), счета на юрлица.
10. **Growth-инструменты** — фолбэк-отчёты → адресные письма дилерам и производителям (продажа факта), лиды из поисковых запросов базы знаний, бейджи «проверенная связка» как совместный маркетинг вендоров, аналитика для будущих ролей (кто из неподключённых сторон чаще всего всплывает в entitlement — тот следующий в воронке продаж).
11. **Комплаенс и доверие** — изоляция данных вендоров, агрегированная сравнительная аналитика только с согласия, политика диагностической утилиты (ручной запуск, явное согласие), заготовка on-premise контура (парсинг и инференс в периметре клиента) как ответ на «документация не должна покидать контур» — продаётся, но не строится до явного спроса.

## 6. Взаимосвязи одним взглядом

Профиль установки питает агента контекстом → агент закрывает 50–70% потока и производит квалифицированную диагностику → entitlement превращает квалификацию в однозначного адресата из трёх внешних ролей → решённые тикеты возвращаются в базу знаний и дообучают агента (доля закрытого растёт сама) → база знаний тянет SEO-воронку новых клиентов → фолбэки размечают, кого из дилеров/интеграторов звать следующими → каждая новая роль наполняет entitlement и расширяет покрытие → покрытие продаёт клиентам более дорогие SLA-тарифы → выручка. Все контуры замкнуты; единственная несущая техническая ставка — качество закрытия агентом, и все инструменты суперадмина в конечном счёте обслуживают её.

## 7. MVP-срез (что из этого строится первым)

Фаза пилота: профиль установки + entitlement как поля (без реестров и верификации), агент по 2 вендорам (vision-пайплайн + золотые наборы), тикеты с ручной маршрутизацией и еженедельным отчётом вендору вместо живого дашборда, публичная база знаний, зона поддержки как YAML. Отложено до платных клиентов: партнёрский и интеграторский кабинеты (роли живут как поля entitlement и фолбэк-тексты), биржа, паспорт проекта как услуга «руками», стенд-автоматизация (воспроизведение вручную), 24/7, публичный рейтинг, on-premise.
