# ASUTPORT — единый roadmap

Master-последовательность фаз. При конфликте с другими файлами — приоритет у этого документа
и `asutport-product-directives.mdc`. Логика порядка унаследована от MAAS: биллинг и лимиты ДО
основной функциональности, суперадминка после стабильных событий, production-hardening финалом.

---

## Фаза 1 — Скелет и инфраструктура

- Docker Compose: nginx, backend, frontend, wordpress, postgres (pgvector), mysql, **minio (dev)**.
- Наружу только nginx 80/443. Роутинг: `/` → WP, `/kb` + `/dashboard` + `/vendor` + `/admin` → Next,
  `/api` + `/health` → Go. Next БЕЗ basePath.
- Go: health (`postgres`, `s3` статусы), chi, pgx, goose при старте, S3-клиент (`internal/s3`),
  миграции 001 (users, organizations, org_members, api_keys).
- Next.js 15: route-группы (public)/(dashboard)/(vendor)/(admin), тёмная HMI-тема из
  `design/asutport-prototype.html`, шрифты через next/font.
- .env.example (включая S3_*), Makefile, скрипты deploy/backup/ssl по deploy-правилам.

**Критерий:** `make dev`; `curl /health` → `{"status":"ok","postgres":"ok","s3":"ok"}`;
`/dashboard` открывается в тёмной теме; корень — WP wizard; в MinIO создаётся бакет.

## Фаза 2 — Аутентификация, организации, роли

- Регистрация/логин, JWT с org_id + role. Организации и членства.
- Middleware org-скоупинга (КАЖДЫЙ запрос данных — через org-контекст).
- Суперадмин — seed/CLI. API-ключи (для будущей диагностической утилиты и интеграций).

## Фаза 3 — Биллинг (фундамент)

- plans (per org_type, ticket_quota, sla_matrix), subscriptions, payments, service_orders.
- Проверка квоты тикетов (emergency не блокируется). Инвойс PDF в S3. Ручная фиксация оплаты.
- CRUD тарифов в API (+ задел под админ-UI).

## Фаза 4 — Документация: vision-пайплайн и S3

- doc_sources/doc_chunks, загрузка PDF (админка/CLI) → S3 → воркеры: render → VLM parse → embed.
- Content-hash идемпотентность. Статусы и ретраи, отчёт об ошибках парсинга.
- Presigned URL endpoint страниц с проверкой прав.
- **Golden sets:** формат YAML, `make golden`, первые наборы на пилотной документации.

**Критерий:** мануал 300 стр. проходит пайплайн; таблица регистров ищется и открывается PNG страницы.

## Фаза 5 — Агент и публичная база знаний

- Чат агента (стриминг): ретривал с фильтрами версии, синтез с citations, валидатор citations.
- Квалификация SGR (тип/приоритет/продукты), уточняющие вопросы, честный отказ.
- /kb: SSR-статьи, SEO-мета, виджет агента (ограниченный без регистрации).
- Профиль установки: визард + подтверждение дельты в чате.

## Фаза 6 — Тикеты, маршрутизация, SLA

- tickets + ticket_events (append-only, конечный автомат), вложения в S3.
- Entitlement-маршрутизация + фолбэки + fallback_log. Support zones (YAML-импорт) в квалификации.
- SLA: таймеры, watchdog-воркер, автоэскалация, ball_owner. Стыковые: связанные тикеты (асимметричный режим).
- Кабинет клиента: тикеты/деталка/таймеры. Кабинет вендора: очередь эскалаций, ответ, передача.

**Критерий:** сквозной сценарий прототипа воспроизводится живьём: чат → эскалация → ответ вендора →
мяч → SLA закрыт → черновик статьи.

## Фаза 7 — Контур знаний и качества

- kb_draft воркер (тикет → анонимизированный черновик → утверждение вендором → публикация).
- qualification_feedback + еженедельный отчёт точности.
- Ежемесячный отчёт вендору (генерируемый).

## Фаза 8 — Суперадминка и операционка

- Дашборд метрик здоровья: доля закрытого агентом, фолбэки, точность квалификации, время реакции
  вендоров, MRR/маржинальность (usage_log).
- Онбординг вендора end-to-end в UI: документация → пайплайн → golden → зона поддержки.
- Организации/пользователи: блокировки, impersonate (audit), смена тарифов. Арбитраж стыковых.

## Фаза 9 — Уведомления и polish

- Email + in-app по событиям домена (тикеты, SLA, квоты, отчёты). Дизайн-полировка по прототипу.
- Онбординг-пустые состояния, инвойсы, документация для клиентов в /kb.

## Фаза 10 — Production-hardening

- NATS JetStream (воркеры), Dragonfly (кэш ретривала), rate limits по тарифам.
- SSL (scripts/ssl-init.sh), мониторинг, бэкапы + s3-sync, логи.
- Нагрузочное: 100 одновременных чатов агента; деградация OpenRouter — честные фолбэки.

## Cross-cutting сводка

| Тема | Когда |
|------|-------|
| S3 (MinIO dev / Beget prod) | Фаза 1, используется везде дальше |
| Org-скоупинг | Фаза 2, обязателен во всех последующих |
| Квоты тикетов | Фаза 3, проверка в Фазе 6 |
| Golden sets | Фаза 4, в CI с Фазы 5 |
| fallback_log | Фаза 6 |
| Партнёр/интегратор/биржа/стенд | Бэклог, не в фазах |
